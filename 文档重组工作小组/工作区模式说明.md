## 一、模式概述

**工作区模式**是一种隔离式任务管理方法，通过创建独立的工作空间来完成特定事务，待事务结束后再合并到主项目。

---

## 二、核心思想

```
主项目/                          # 生产环境（稳定）
├── 代码/
├── 文档/
├── 配置/
└── workspace-{日期}-{任务}/     # 工作区（隔离开发）
    ├── 方案.md
    ├── 规范.md
    ├── 执行脚本
    └── 完成报告.md
```

类似于：
- Git 的 feature branch
- Docker 的容器
- 数据库的事务
- 软件开发的沙盒

---

## 三、模式优势

### 3.1 隔离性
- 工作区独立于主项目，不污染生产环境
- 可以大胆尝试、修改，无后顾之忧
- 事务失败时可整体丢弃，零风险

### 3.2 可追溯
- 完整记录思考过程和决策依据
- 方案演变有迹可循
- 便于后期复盘和知识沉淀

### 3.3 协作友好
- 工作进度透明可见
- 方案文档化，便于评审
- 合并前可充分验证

### 3.4 风险控制
- 确认无误后再合并
- 有完整的回滚方案
- 降低生产环境风险

---

## 四、标准结构

```
workspace-YYYYMMDD-{任务名}/
├── README.md              # 工作区说明
├── 方案.md                 # 详细方案设计
├── 规范.md                 # 执行规范/标准
├── 执行脚本.*              # 自动化脚本（如有）
├── 验证清单.md             # 检查清单
└── 完成报告.md             # 总结报告
```

---

## 五、生命周期

```
创建 → 规划 → 执行 → 验证 → 合并 → 归档
```

| 阶段 | 动作 | 输出 |
|:---|:---|:---|
| 创建 | 建立工作区目录 | workspace/ |
| 规划 | 编写方案和规范 | 方案.md, 规范.md |
| 执行 | 实施具体任务 | 执行结果 |
| 验证 | 按清单检查 | 验证报告 |
| 合并 | 应用到主项目 | 主项目更新 |
| 归档 | 保留或删除工作区 | 历史记录 |

## 七、使用场景

### 适用场景
- ✅ 大型文档重组
- ✅ 代码重构
- ✅ 数据库迁移
- ✅ 架构升级
- ✅ 配置变更
- ✅ 任何有风险的批量操作

### 不适用场景
- ❌ 简单的单行修改
- ❌ 紧急热修复
- ❌ 已经充分验证的例行操作

---

## 八、最佳实践

### 8.1 创建时
- 明确任务边界和目标
- 制定清晰的验收标准
- 准备回滚方案

### 8.2 执行时
- 所有操作在工作区内完成
- 及时记录决策和变更
- 保持与主项目的同步

### 8.3 合并时
- 使用 checklist 逐项验证
- 小步提交，便于回滚
- 保留工作区作为历史记录

## 十、总结

工作区模式是一种**工程化管理思维**，通过隔离、文档化、流程化来降低风险、提高效率。
