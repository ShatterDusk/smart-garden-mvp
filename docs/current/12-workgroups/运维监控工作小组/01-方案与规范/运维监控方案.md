## 一、监控目标

### 1.1 核心目标
1. **系统稳定性**: 实时监测系统健康状态，及时发现异常
2. **性能优化**: 识别性能瓶颈，提供优化建议
3. **故障预警**: 在问题影响用户前发现并处理
4. **日志分析**: 通过日志追踪问题根因

### 1.2 监控范围
| 层级 | 监控对象 | 数据来源 |
|------|----------|----------|
| 前端 | 微信小程序 | `client_logs` 表 |
| 后端 | Node.js/Express | `system_logs` 表 |
| 数据库 | MySQL | 性能指标查询 |
| 业务 | API接口、定时任务 | 日志 + 自定义指标 |

## 二、监控路径设计

### 路径1: 系统错误监控 🔴 P0
**目标**: 发现并处理系统级错误

**监控内容**:
- 后端错误日志 (`level = 'error' OR 'fatal'`)
- 前端错误日志 (`level = 'error' OR 'fatal'`)
- 异常堆栈分析
- 错误趋势分析

**关键指标**:
- 错误率 (错误数/总请求数)
- 错误类型分布
- 新增错误类型
- 高频错误来源

---

### 路径2: API性能监控 🟠 P1
**目标**: 监控API响应性能

**监控内容**:
- API响应时间
- 慢查询日志
- 高并发接口表现
- 请求成功率

**关键指标**:
- P50/P95/P99 响应时间
- 吞吐量 (QPS)
- 错误率
- 超时率

---

### 路径3: 数据库性能监控 🟠 P1
**目标**: 监控数据库健康状态

**监控内容**:
- 慢查询分析
- 连接池状态
- 表空间使用
- 索引效率

**关键指标**:
- 慢查询数量
- 平均查询时间
- 连接数使用率
- 锁等待时间

---

### 路径4: 业务异常监控 🟡 P2
**目标**: 发现业务逻辑异常

**监控内容**:
- 用户行为异常
- 业务流程中断
- 数据一致性检查
- 定时任务执行状态

**关键指标**:
- 业务流程完成率
- 异常操作次数
- 任务失败率
- 数据异常条目数

---

### 路径5: 资源使用监控 🟡 P2
**目标**: 监控系统资源使用

**监控内容**:
- 服务器资源 (CPU/内存/磁盘)
- 存储空间 (COS/数据库)
- 网络流量
- 第三方服务状态

**关键指标**:
- CPU使用率
- 内存使用率
- 磁盘使用率
- COS存储使用率

---

### 路径6: 安全事件监控 🔴 P0
**目标**: 发现安全威胁

**监控内容**:
- 异常访问模式
- 认证失败事件
- 敏感操作日志
- 潜在攻击行为

**关键指标**:
- 认证失败次数
- 异常IP访问
- 敏感接口调用频率
- 权限越界尝试

## 三、数据源说明

### 3.1 后端日志表 (`system_logs`)
| 字段 | 说明 | 监控用途 |
|------|------|----------|
| `level` | 日志级别 | 错误筛选 |
| `message` | 日志内容 | 问题分析 |
| `source` | 来源模块 | 问题定位 |
| `request_id` | 请求追踪ID | 链路追踪 |
| `url` | 请求URL | API分析 |
| `method` | HTTP方法 | 接口分析 |
| `error_stack` | 错误堆栈 | 根因分析 |
| `created_at` | 创建时间 | 时间序列分析 |

### 3.2 前端日志表 (`client_logs`)
| 字段 | 说明 | 监控用途 |
|------|------|----------|
| `user_id` | 用户ID | 用户问题追踪 |
| `session_id` | 会话ID | 会话分析 |
| `level` | 日志级别 | 错误筛选 |
| `page_path` | 页面路径 | 页面问题定位 |
| `action` | 用户行为 | 行为分析 |
| `device_info` | 设备信息 | 设备兼容性分析 |
| `network_type` | 网络类型 | 网络问题分析 |

## 四、告警策略

### 4.1 告警级别定义

| 级别 | 触发条件 | 响应时间 | 通知方式 |
|------|----------|----------|----------|
| 🔴 P0-紧急 | 系统不可用/数据丢失 | 立即 | 立即通知 |
| 🟠 P1-高 | 功能受损/性能严重下降 | 15分钟内 | 及时通知 |
| 🟡 P2-中 | 非核心功能异常 | 1小时内 | 汇总通知 |
| 🟢 P3-低 | 优化建议/轻微异常 | 24小时内 | 日报汇总 |

### 4.2 告警规则示例

```yaml
# P0 告警规则
- name: 后端错误激增
  condition: 5分钟内 error 级别日志 > 50条
  level: P0
  
- name: 前端大量报错
  condition: 5分钟内 client error 日志 > 20条
  level: P0

# P1 告警规则
- name: API响应缓慢
  condition: P95响应时间 > 2000ms 持续5分钟
  level: P1

- name: 数据库慢查询
  condition: 慢查询数量 > 10条/分钟
  level: P1

# P2 告警规则
- name: 定时任务失败
  condition: 任务失败率 > 10%
  level: P2
```

## 五、监控工具

### 5.1 内置工具
- **MCP MySQL**: 直接查询日志数据库
- **日志查询脚本**: 预定义的常用查询
- **趋势分析**: 基于时间序列的日志统计

### 5.2 常用查询模板

```sql
-- 查询最近错误
SELECT level, message, source, created_at 
FROM system_logs 
WHERE level IN ('error', 'fatal') 
ORDER BY created_at DESC LIMIT 50;

-- 按模块统计错误
SELECT source, COUNT(*) as count 
FROM system_logs 
WHERE level = 'error' 
  AND created_at >= DATE_SUB(NOW(), INTERVAL 1 HOUR)
GROUP BY source 
ORDER BY count DESC;

-- 追踪请求链路
SELECT * FROM system_logs 
WHERE request_id = 'xxx' 
ORDER BY created_at ASC;
```
