## 一、监控执行标准

### 1.1 监控频率

| 监控路径 | 检查频率 | 数据时间窗口 | 优先级 |
|----------|----------|--------------|--------|
| 路径1: 系统错误监控 | 每15分钟 | 最近1小时 | P0 |
| 路径2: API性能监控 | 每30分钟 | 最近2小时 | P1 |
| 路径3: 数据库性能监控 | 每小时 | 最近4小时 | P1 |
| 路径4: 业务异常监控 | 每小时 | 最近4小时 | P2 |
| 路径5: 资源使用监控 | 每天 | 最近24小时 | P2 |
| 路径6: 安全事件监控 | 每15分钟 | 最近1小时 | P0 |

### 1.2 数据查询限制
- **单次查询记录数**: ≤ 1000 条
- **单次查询时间范围**: ≤ 24 小时
- **并发查询数**: ≤ 3 个
- **查询超时**: 30 秒

## 二、问题严重等级定义

### 🔴 P0-紧急（Critical）
**定义**: 系统不可用或核心功能完全失效

**触发条件**:
- 后端错误率 > 10%
- 数据库连接失败
- 关键API完全不可用
- 数据丢失或损坏
- 安全入侵事件

**处理要求**:
- 立即响应（5分钟内）
- 通知相关负责人
- 启动应急响应流程
- 持续跟踪直至解决

---

### 🟠 P1-高（High）
**定义**: 功能受损或性能严重下降

**触发条件**:
- API P95响应时间 > 2000ms
- 错误率 1% ~ 10%
- 数据库慢查询激增
- 定时任务连续失败
- 异常访问模式

**处理要求**:
- 15分钟内响应
- 记录问题详情
- 制定修复计划
- 4小时内给出解决方案

---

### 🟡 P2-中（Medium）
**定义**: 非核心功能异常或轻微性能问题

**触发条件**:
- 偶发性错误（< 1%）
- API响应时间轻微升高
- 资源使用率超过阈值
- 业务流程偶发中断

**处理要求**:
- 1小时内响应
- 评估影响范围
- 在下次迭代中修复

---

### 🟢 P3-低（Low）
**定义**: 优化建议或轻微异常

**触发条件**:
- 代码优化建议
- 日志规范问题
- 轻微性能优化空间
- 监控盲区发现

**处理要求**:
- 记录在案
- 方便时处理

## 三、监控执行流程

### 3.1 单路径监控流程

```
1. 准备阶段
   ├── 阅读该路径的监控清单
   ├── 准备查询语句
   └── 准备问题记录模板

2. 数据采集阶段
   ├── 执行预定义查询
   ├── 收集监控指标
   └── 保存原始数据

3. 分析阶段
   ├── 对比历史基线
   ├── 识别异常模式
   ├── 评估影响范围
   └── 确定问题等级

4. 报告阶段
   ├── 记录发现的问题
   ├── 输出监控报告
   ├── 更新问题清单
   └── 通知相关人员（如需要）
```

### 3.2 问题记录规范

#### 必填字段
```yaml
问题编号: OM-{路径编号}-{序号}  # 例如: OM-1-001
监控路径: 路径1-系统错误监控
严重程度: 🔴P0 / 🟠P1 / 🟡P2 / 🟢P3
发现时间: YYYY-MM-DD HH:mm:ss
问题类型: 错误/性能/安全/资源/业务
问题描述: 具体描述问题现象
影响范围: 影响的用户/功能/模块
根因分析: 初步分析的问题原因
处理建议: 具体的处理方案
状态: 待处理/处理中/已解决/已忽略
```

#### 可选字段
```yaml
相关日志: 关联的日志ID或查询条件
复现步骤: 如何复现该问题
责任人: 负责处理的人员
处理时间: YYYY-MM-DD HH:mm:ss
验证人: 验证解决的人员
备注: 其他补充信息
```

## 六、工具使用规范

### 6.1 推荐工具

| 工具 | 用途 | 使用场景 |
|------|------|----------|
| MCP MySQL | 数据库查询 | 日志查询、性能分析 |
| SQL脚本 | 预定义查询 | 常用监控查询 |
| 日志分析 | 文本分析 | 错误堆栈分析 |

### 6.2 常用查询模板

```sql
-- 查询最近错误日志
SELECT id, level, message, source, request_id, created_at
FROM system_logs
WHERE level IN ('error', 'fatal')
  AND created_at >= DATE_SUB(NOW(), INTERVAL 1 HOUR)
ORDER BY created_at DESC
LIMIT 100;

-- 按模块统计错误
SELECT source, level, COUNT(*) as count
FROM system_logs
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 1 HOUR)
GROUP BY source, level
ORDER BY count DESC;

-- 追踪请求链路
SELECT id, level, message, source, created_at
FROM system_logs
WHERE request_id = ?
ORDER BY created_at ASC;

-- 查询前端错误
SELECT id, user_id, page_path, level, message, created_at
FROM client_logs
WHERE level IN ('error', 'fatal')
  AND created_at >= DATE_SUB(NOW(), INTERVAL 1 HOUR)
ORDER BY created_at DESC
LIMIT 100;

-- 慢查询分析（需开启慢查询日志）
SELECT * FROM mysql.slow_log
WHERE start_time >= DATE_SUB(NOW(), INTERVAL 1 HOUR)
ORDER BY query_time DESC
LIMIT 50;
```

## 七、协作规范

### 7.1 日常监控
- 按频率执行各路径监控
- 及时更新问题清单状态
- 记录监控执行日志

### 7.2 问题升级
当发现以下情况时，立即升级：
- P0级别问题
- 同类问题频繁发生（>3次/天）
- 问题影响范围扩大
- 根因不明的问题

### 7.3 文档更新
- 监控后立即更新执行记录
- 问题解决后更新问题清单
- 定期输出监控报告
