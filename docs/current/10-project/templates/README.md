# 工作小组模板库

## 概述

本目录包含创建新工作小组的标准模板和工具，确保所有工作小组遵循统一的结构和规范。

---

## 快速开始

### 方法一：使用 PowerShell 脚本（推荐）

```powershell
# 进入模板目录
cd docs/current/10-project/templates/workgroup-template

# 执行创建脚本
.\create-workgroup.ps1 -Name "任务名" -Type "项目型"

# 示例：创建一个 API 重构工作小组
.\create-workgroup.ps1 -Name "API重构" -Type "项目型"
```

**支持的类型**：
- `项目型` - 有明确交付物的项目（一次性）
- `分析型` - 分析类任务（一次性）
- `运维型` - 持续性运维工作（持续性）
- `支撑型` - 支撑性工作（持续性）

### 方法二：手动复制模板

1. 复制 `workgroup-template` 目录
2. 重命名为 `{任务名}工作小组`
3. 替换所有模板中的占位符：
   - `{任务名}` → 实际任务名称
   - `YYYY-MM-DD` → 当前日期
   - `[方括号内容]` → 实际内容

## 模板使用指南

### 1. README.md 模板

**用途**：工作小组入口文档

**必填内容**：
- 工作区类型
- 活动状态
- 工作范围
- 执行路径
- 当前工作状态

**更新频率**：每日更新

### 2. 方案模板

**用途**：详细方案设计

**必填章节**：
- 背景与目标
- 范围与边界
- 总体设计
- 详细设计
- 执行计划
- 风险评估

**完成标准**：方案评审通过

### 3. 执行规范模板

**用途**：定义执行标准和流程

**必填章节**：
- 执行标准（质量/命名）
- 工作流程
- 工具使用规范
- 沟通规范
- 检查清单

**完成标准**：规范评审通过

### 4. 执行记录模板

**用途**：单路径执行过程记录

**必填内容**：
- 任务目标
- 执行步骤
- 关键决策
- 问题记录
- 执行结果

**完成标准**：路径执行完成

### 5. 问题清单模板

**用途**：问题跟踪和管理

**必填内容**：
- 问题统计
- 问题列表（按优先级）
- 问题趋势
- 升级机制

**更新频率**：实时更新

### 6. 完成报告模板

**用途**：工作小组关闭时总结

**必填章节**：
- 执行摘要
- 目标达成情况
- 经验总结
- 知识沉淀
- 后续工作

**完成标准**：报告审批通过

## 占位符说明

| 占位符 | 含义 | 示例 |
|:---|:---|:---|
| `{任务名}` | 工作小组的任务名称 | API重构、静态分析 |
| `{项目型/分析型/运维型/支撑型}` | 工作小组类型 | 项目型 |
| `{一次性/持续性/阶段性/响应式}` | 工作模式 | 一次性 |
| `{序号}` | 路径或问题序号 | 1、2、3 |
| `{前缀}` | 问题来源前缀 | OM、SA、DEV |
| `YYYY-MM-DD` | 日期格式 | 2026-04-11 |
| `HH:MM` | 时间格式 | 14:30 |
| `[内容]` | 需要填写的内容 | [根据实际情况填写] |

---

## 工作小组类型说明

| 类型 | 特点 | 生命周期 | 适用场景 |
|:---|:---|:---|:---|
| **项目型** | 有明确交付物 | 创建→规划→执行→验证→合并→关闭 | 功能开发、重构 |
| **分析型** | 产出分析报告 | 创建→规划→执行→报告→归档 | 代码审查、性能分析 |
| **运维型** | 持续性工作 | 创建→规划→持续执行→定期报告 | 监控、维护 |
| **支撑型** | 支撑性工作 | 创建→规划→持续运营 | 知识管理、工具开发 |

## 最佳实践

### 创建时
1. 明确工作小组类型
2. 填写完整的元信息
3. 定义清晰的工作范围
4. 识别依赖关系

### 执行时
1. 每日更新活动状态
2. 及时记录执行过程
3. 实时跟踪问题
4. 保持文档同步

### 关闭时
1. 完成所有路径
2. 清理问题清单
3. 编写完成报告
4. 沉淀知识到知识库
