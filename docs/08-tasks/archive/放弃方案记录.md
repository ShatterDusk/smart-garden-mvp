# 放弃方案记录

本文档记录经过讨论后决定不实现的方案，仅作参考。

---

## 放弃方案列表

### 1. messages.is_read 字段

**提出时间**: 2026-04-02  
**放弃时间**: 2026-04-02  
**放弃原因**: 数据冗余，多设备同步困难

#### 原始方案
```sql
ALTER TABLE messages 
ADD COLUMN is_read TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否已读：0-未读，1-已读';
```

#### 问题分析
1. **数据冗余**: 消息本身没有"已读"概念，是"用户-消息"的关系
2. **多设备同步困难**: 手机读了，平板应该同步显示已读，单字段无法满足
3. **扩展性差**: 如果以后需要"已读时间"、"最后阅读位置"等，单字段不够

#### 替代方案
**user_config JSON 存储方案**（零数据库变更）：
```javascript
{
  "read_positions": {
    "session_001": "MSG_123456",
    "session_002": "MSG_789012"
  }
}
```

### 2. health_scores 表

**提出时间**: 2026-04-02  
**放弃时间**: 2026-04-02  
**放弃原因**: 数据冗余，可用 diagnosis_cards 查询替代

#### 原始方案
```sql
CREATE TABLE health_scores (
  score_id VARCHAR(64) NOT NULL COMMENT '唯一标识，UUID',
  plant_id VARCHAR(64) NOT NULL COMMENT '关联植物',
  health_score INT NOT NULL COMMENT '健康评分 0-100',
  status ENUM('healthy','warning','critical') NOT NULL COMMENT '健康状态',
  source_type ENUM('diagnosis','manual','system') NOT NULL DEFAULT 'diagnosis' COMMENT '评分来源',
  source_id VARCHAR(64) NULL COMMENT '来源ID，如诊断卡ID',
  recorded_at DATETIME NOT NULL COMMENT '评分记录时间',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (score_id),
  INDEX idx_plant_time (plant_id, recorded_at),
  INDEX idx_plant_status (plant_id, status),
  CONSTRAINT fk_health_scores_plant FOREIGN KEY (plant_id) REFERENCES plants(plant_id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='健康评分历史表';
```

#### 问题分析
1. **数据冗余**: diagnosis_cards 表已经包含 health_score 字段
2. **维护成本**: 需要额外维护一张表，增加系统复杂度
3. **同步问题**: 诊断卡和健康评分数据可能不一致

#### 替代方案
**直接查询 diagnosis_cards 表**：
```sql
SELECT 
  DATE(created_at) as date,
  health_score,
  status
FROM diagnosis_cards 
WHERE plant_id = ? 
  AND health_score IS NOT NULL
ORDER BY created_at DESC 
LIMIT 30;
```

### 3. message_reads 表

**提出时间**: 2026-04-02  
**放弃时间**: 2026-04-02  
**放弃原因**: 过度设计，AI 会话不需要精确已读状态

#### 原始方案
```sql
CREATE TABLE message_reads (
  read_id VARCHAR(64) NOT NULL COMMENT '唯一标识，UUID',
  message_id VARCHAR(64) NOT NULL COMMENT '关联消息',
  user_id VARCHAR(64) NOT NULL COMMENT '阅读用户',
  read_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '阅读时间',
  PRIMARY KEY (read_id),
  UNIQUE KEY uk_message_user (message_id, user_id),
  INDEX idx_user_time (user_id, read_at),
  CONSTRAINT fk_message_reads_message FOREIGN KEY (message_id) REFERENCES messages(message_id) ON DELETE CASCADE,
  CONSTRAINT fk_message_reads_user FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='消息已读记录表';
```

#### 问题分析
1. **过度设计**: AI 会话只需要红点提示，不需要精确到单条消息的已读状态
2. **数据量大**: 每条消息的每个用户都要记录，数据量巨大
3. **查询复杂**: 需要关联查询判断未读状态

#### 替代方案
**user_config JSON 存储方案**（零数据库变更）：
```javascript
{
  "read_positions": {
    "session_001": "MSG_123456",
    "session_002": "MSG_789012"
  }
}
```

## 放弃原则总结

| 原则 | 说明 |
|:---|:---|
| 避免数据冗余 | 能用查询解决的，不新建表 |
| 避免过度设计 | 根据业务需求选择合适方案 |
| 零变更优先 | 能不修改数据库 schema 的，优先不改 |
| 简单优先 | 简单方案能满足需求的，不用复杂方案 |
