## 演示流程

### 【0:00-1:00】开头：项目起源

**屏幕**: GitHub仓库首页

**话术**:
> "这个项目其实是挑战杯的参赛项目，后来没继续参赛。
> 
> 题目不是我想的，但怎么做是我来设计的——会话怎么组织、诊断卡长什么样、数据怎么流动，这些细节都是我定的。
> 
> 灵感确实是养多肉的时候来的——当时想，如果拍照就能知道植物怎么了就好了。
> 
> 后来觉得投了不少设计成本，丢了可惜，就自己做下去了。"

**动作**: 滚动GitHub README，展示项目概览

---

### 【1:00-3:00】产品演示

**屏幕**: 切换到微信开发者工具

**话术**:
> "先看看做了个什么东西。
> 
> 这是一个植物医生——拍照能诊断植物问题，也可以问AI养护建议。"

**操作路径**:
```
首页 → 点击"我的植物" → 植物列表 → 点击某个植物 → 植物详情
```

**话术**（边操作边说）:
> "用户登录后看到首页，可以添加植物档案。
> 
> 点进一株植物，可以看到基本信息、环境数据、诊断历史。"

**操作**: 点击"AI诊断"按钮

> "这是核心功能——AI诊断。上传一张植物照片...
> 
> 系统调用智谱GLM进行识别和诊断，生成诊断卡片——健康分数、发现的问题、养护建议。"

**操作**: 返回首页 → 点击"AI助手"

> "用户也可以直接和AI对话，咨询养护问题。会话分两种：咨询会话不绑定植物，植物会话绑定特定植物。"

### 【3:00-5:00】技术架构

**屏幕**: 切换到浏览器 → `mermaid图表.md`

**话术**:
> "来看看背后是怎么实现的。"

**操作**: 滚动到"系统架构图（整体）"

**话术**:
> "这是整体架构。微信小程序 + Node.js后端 + MySQL数据库。
> 
> 重点看这几个地方：
> - **中间件层**：用户请求走auth.js，设备数据走deviceAuth.js，两个入口分开
> - **服务层**：所有Service继承BaseService，复用CRUD方法
> - **外部服务**：调用智谱GLM做AI分析，和风天气做数据补偿"

**操作**: 滚动到"环境数据补偿流程"

> "这是数据补偿的逻辑——当传感器离线时，定时任务用天气API补偿数据，标记is_stale=true。等传感器重新上线，补传数据会覆盖补偿数据。"

### 【5:00-9:00】亮点1：AI辅助开发

**屏幕**: 切换到VS Code → `BaseService.js`

**话术**:
> "聊聊怎么做的。我用AI辅助开发，这个项目大概3周写完，27000行代码。
> 
> 来看几个AI写的代码——"

**操作**: 展示 `BaseService.js`

> "这是BaseService，所有Service的基类。AI帮我生成了CRUD方法——findById、create、update、delete、paginate。
> 
> 我只需要告诉它：'写一个BaseService，包含CRUD和分页'，它就生出来了。
> 
> 然后UserService、PlantService继承它，自动拥有这些方法。"

**操作**: 切换到 `EnvironmentService.js`

> "这个是数据补偿的核心逻辑。我告诉AI：'当传感器数据上报时，检查是否有补偿数据，有的话就覆盖'。
> 
> AI生成了这个switch-case结构——pending、received、compensated三种状态的处理。
> 
> 当然，不是一次就对的。我改了好几轮，跟AI说'这里逻辑不对'、'那个边界情况没考虑'。"

**操作**: 切换到浏览器 → `代码演示片段.md`

**话术**:
> "这是几个关键代码片段，演示时可以直接看。
> 
> **AI辅助开发的体会**：
> 1. AI擅长写样板代码——CRUD、参数校验、错误处理
> 2. AI不擅长理解业务——数据补偿的状态机是我画图跟它解释的
> 3. 要会提问——'写一个XX'不如'写一个XX，要求A、B、C'
> 4. 要会审查——AI写的代码要自己看懂，不然出bug找不到"

### 【9:00-12:00】亮点2：GitHub二次开发

**屏幕**: 切换到浏览器 → GitHub原项目页面

**话术**:

> 我需要它直接往我的后端API发数据，所以做了二次开发。"

**屏幕**: 切换到VS Code → `http_helper_v2.py`

**话术**:
> "我加了这个HTTP推送模块。

> 
> 这个补传机制是我自己设计的，AI帮写代码，但逻辑是我定的。

> **二次开发的体会**：
> 1. 先读懂原代码——我花了一天看它的架构

**屏幕**: 切换到浏览器 → `虚拟设备操作指南.md`

> "这是我写的使用指南，记录了怎么启动、怎么配置、怎么和后端对接。"

### 【12:00-14:00】亮点3：分层架构

**屏幕**: 切换到VS Code → `app.js` + `routes/index.js`

**话术**:
> "最后讲一下分层架构，这个比较基础但很重要。
> 
> 后端分三层：
> - **Controller**：参数校验、调用Service、格式化响应
> - **Service**：业务逻辑、事务管理、跨模块协调
> - **Model**：数据访问、关联定义"

**操作**: 展示一个Controller → Service → Model的调用链

> "比如用户登录：
> - Controller收到请求，校验参数
> - 调用UserService.login
> - UserService调用User Model查数据库
> - 返回结果，Controller格式化响应"

**话术**:
> "分层的好处：
> 1. 职责清晰——改业务逻辑只改Service，改数据结构只改Model
> 2. 方便测试——可以单独测Service，不依赖数据库
> 3. AI更好理解——告诉它'在Service层加个方法'，它就知道在哪写"
