# 教训总结案例

> 演讲时长：2-3分钟
> 目标受众：计算机/软件专业同学
> 核心信息：真实的踩坑经历、不写文档的后果、不关注架构的代价

***

## 案例一：不写API文档的后果（45秒）⭐重点

### 背景

项目初期，我沉浸在Vibe Coding的高效中，觉得"代码即文档"，没有写API文档。

### 问题发生

前端同学（也就是我自己切到前端视角）调用用户登录接口：

**后端期望的参数**：

```javascript
// POST /api/users/login
{
  "code": "wx_login_code_from_wechat"
}
```

**前端实际传的参数**：

```javascript
{
  "wxCode": "wx_login_code_from_wechat"
}
```

### 后果

1. **调试时间**：花了2小时定位问题
   - 以为是微信登录流程有问题
   - 检查网络请求、查看日志、加断点
   - 最后发现是参数名不匹配
2. **代码混乱**：
   - 有的接口用`code`，有的用`wxCode`
   - 有的用`userId`，有的用`user_id`
   - 命名不一致导致更多bug
3. **返工成本**：
   - 统一修改前端参数名
   - 测试所有相关接口
   - 总计耗时4小时

### 修复措施

1. **强制文档先行**：
   ```
   改API → 先更新文档 → 再改代码
   ```
2. **建立规范**：
   - 所有API必须在`docs/02-architecture/API接口设计.md`中定义
   - 参数命名统一使用camelCase
   - 响应格式统一`{code, message, data}`
3. **代码校验**：
   - 使用Joi进行参数校验
   - 不合法的参数直接返回400错误

### 教训

> "省下的10分钟文档时间，会变成2小时的调试时间。
> 文档不是写给别人的，是写给未来的自己。"

## 案例二：不关注架构的代价（45秒）⭐重点

### 背景

项目初期，为了快速验证想法，我把所有逻辑都写在Controller里。

### 代码现状（反例）

```javascript
// sessionController.js - 早期版本
async createMessage(req, res) {
  // 1. 参数校验
  const { sessionId, content, type } = req.body;
  if (!sessionId || !content) {
    return res.status(400).json({ error: '参数缺失' });
  }
  
  // 2. 查询会话
  const session = await Session.findByPk(sessionId);
  if (!session) {
    return res.status(404).json({ error: '会话不存在' });
  }
  
  // 3. 查询植物信息
  const plant = await Plant.findByPk(session.plantId);
  
  // 4. 查询环境数据
  const environment = await EnvironmentReading.findOne({
    where: { plantId: plant.plantId },
    order: [['createdAt', 'DESC']]
  });
  
  // 5. 组装Prompt
  const prompt = `植物：${plant.nickname}...`;
  
  // 6. 调用AI
  const aiResponse = await callAI(prompt);
  
  // 7. 保存消息
  const message = await Message.create({
    sessionId,
    role: 'user',
    content,
    // ... 更多字段
  });
  
  // 8. 保存AI回复
  await Message.create({
    sessionId,
    role: 'assistant',
    content: aiResponse,
    // ... 更多字段
  });
  
  // 9. 返回响应
  res.json({ code: 0, data: message });
}
```

### 问题

1. **Controller臃肿**：一个方法150行，职责不清
2. **无法测试**：依赖数据库、AI服务，单元测试几乎不可能
3. **代码重复**：其他Controller也有类似的AI调用逻辑
4. **难以维护**：修改AI调用方式，需要改N个地方

### 重构方案

引入Service层：

```
Controller（参数校验、响应格式化）
    ↓
Service（业务逻辑、跨模块协调）
    ↓
Model（数据访问）
```

**重构后**：

```javascript
// sessionController.js - 重构后
async createMessage(req, res) {
  const { sessionId, content, type } = req.body;
  const result = await sessionService.createMessage(sessionId, content, type);
  res.success(result);
}

// SessionService.js
class SessionService {
  async createMessage(sessionId, content, type) {
    // 业务逻辑封装，可独立测试
  }
}
```

### 重构收益

| 指标             |  重构前 |  重构后 |
| :------------- | :--: | :--: |
| Controller平均行数 | 150+ |  30  |
| 单元测试覆盖率        |  20% |  60% |
| 新增功能开发时间       |  2天  | 0.5天 |
| 代码重复率          |  40% |  10% |

### 教训

> "快速验证不等于随意堆砌。
> 即使MVP，也要有基本的架构意识。
> 重构的成本，是前期设计投入的10倍。"

## 案例三：数据模型设计失误（45秒）

### 背景

设计EnvironmentReading表时，没有考虑数据来源的多样性。

### 初始设计（反例）

```javascript
// models/EnvironmentReading.js - 初始版本
const EnvironmentReading = sequelize.define('EnvironmentReading', {
  readingId: { type: DataTypes.STRING(64), primaryKey: true },
  plantId: { type: DataTypes.STRING(64), allowNull: false },
  temperature: DataTypes.FLOAT,
  humidity: DataTypes.FLOAT,
  light: DataTypes.FLOAT,
  // ... 其他字段
  createdAt: DataTypes.DATE
});
```

### 需求变更

后来需要支持：

1. **传感器数据**：设备实时上报
2. **天气API数据**：传感器缺失时补偿

### 问题

1. **无法区分数据来源**：不知道这条数据是传感器还是天气API
2. **无法标记数据质量**：不知道是否是补偿数据（可能过时）
3. **数据库迁移复杂**：
   - 需要添加字段
   - 历史数据需要回填
   - 所有查询逻辑需要修改

### 修复过程

**迁移脚本**：

```javascript
// migrations/xxx-add-data-source.js
module.exports = {
  up: async (queryInterface, Sequelize) => {
    await queryInterface.addColumn('environment_readings', 'data_source', {
      type: Sequelize.ENUM('sensor', 'weather_api', 'manual'),
      defaultValue: 'sensor'
    });
    await queryInterface.addColumn('environment_readings', 'is_stale', {
      type: Sequelize.BOOLEAN,
      defaultValue: false
    });
    // 回填历史数据...
  },
  down: async (queryInterface, Sequelize) => {
    // ...
  }
};
```

**修改范围**：

- 数据模型：2个文件
- 服务层：5个文件
- 控制器：3个文件
- 前端：4个文件
- 测试：10+个文件

**总耗时**：2天

### 如果一开始就设计好

```javascript
// models/EnvironmentReading.js - 正确设计
const EnvironmentReading = sequelize.define('EnvironmentReading', {
  readingId: { type: DataTypes.STRING(64), primaryKey: true },
  plantId: { type: DataTypes.STRING(64), allowNull: false },
  
  // 数据来源追踪（一开始就设计）
  dataSource: {
    type: DataTypes.ENUM('sensor', 'weather_api', 'manual', 'compensation'),
    allowNull: false
  },
  isStale: {
    type: DataTypes.BOOLEAN,
    defaultValue: false,
    comment: '是否为补偿数据（可能不准确）'
  },
  
  // 数据值
  temperature: DataTypes.FLOAT,
  humidity: DataTypes.FLOAT,
  // ...
});
```

### 教训

> "数据模型是系统的根基，根基不稳，地动山摇。
> 设计数据模型时，多问一句：
> '未来可能有哪些变化？数据来源会有几种？'
> 多花1小时设计，省2天重构。"

## 总结（30秒）

### 三个核心教训

| 教训       | 解决方案            | 投入产出比 |
| :------- | :-------------- | :---: |
| 不写文档     | 文档先行，强制规范       |  1:12 |
| 不关注架构    | 引入Service层，职责分离 |  1:10 |
| 数据模型设计草率 | 预留扩展性，考虑未来变化    |  1:16 |

### 一句话总结

> "AI可以帮你写代码，但不能替你思考。
> 架构设计、数据建模、文档编写——
> 这些'思考型'工作，是AI时代程序员的核心竞争力。"
