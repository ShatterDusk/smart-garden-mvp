# 架构讲解脚本

> 演讲时长：3-5分钟
> 目标受众：计算机/软件专业同学
> 核心信息：系统架构、技术选型、模块设计、数据流

---

> 微信小程序 + Node.js后端 + MySQL数据库。
> 接下来我会从整体到局部，逐步展开。"

### 配合动作

- 打开`docs/02-architecture/系统架构设计.md`

## 一、整体架构（1分钟）

### 展示架构图

**打开**：`docs/02-architecture/系统架构设计.md`

**讲解词**：
> "这是我们的系统架构图，分为五层：
> 
> 1. **客户端层**：微信小程序 + 硬件设备
> 2. **网关层**：Nginx做反向代理和负载均衡
> 3. **服务层**：Node.js + Express，按领域划分为5个域
> 4. **数据层**：MySQL主存储 + Redis缓存
> 5. **外部服务**：微信、AI模型、天气API"

### 分层职责

| 层级 | 职责 | 技术组件 |
|:---|:---|:---|
| **客户端层** | 用户交互界面 | 微信小程序原生框架 |
| **网关层** | 请求路由、负载均衡、限流 | Nginx API Gateway |
| **服务层** | 业务逻辑处理 | Node.js + Express |
| **数据层** | 数据持久化、缓存 | MySQL 8.0, Redis |
| **外部服务** | 第三方能力集成 | 微信、AI模型、天气API |

### 核心数据流

```
微信小程序 → Nginx → Express → MySQL/Redis
                ↓
            调用外部服务（微信/AI/天气）
```

## 二、服务层设计（1.5分钟）⭐重点

### 领域划分

**展示**：`docs/02-architecture/系统架构设计.md`中的模块划分图

**讲解词**：
> "服务层按领域驱动设计（DDD）划分为5个域：
> 
> 1. **用户域**：认证、用户管理
> 2. **植物域**：植物档案、养护记录、诊断
> 3. **设备域**：设备绑定、环境数据
> 4. **AI域**：会话管理、AI调用
> 5. **基础设施域**：文件存储、日志"

### 模块职责表

| 模块 | 职责 | 核心功能 | 对应路由 |
|:---|:---|:---|:---|
| **认证模块** | 用户身份验证 | 微信登录、Token生成与验证 | `/api/users/login` |
| **用户模块** | 用户数据管理 | 用户信息CRUD、配置管理 | `/api/users` |
| **植物模块** | 植物档案管理 | 植物CRUD、设备绑定 | `/api/plants` |
| **养护模块** | 养护记录管理 | 养护记录CRUD | `/api/care-records` |
| **诊断模块** | 诊断历史管理 | 诊断记录查询 | `/api/diagnosis` |
| **设备模块** | 硬件设备管理 | 设备绑定、状态监控 | `/api/devices` |
| **环境模块** | 环境数据管理 | 数据上报、历史查询、补偿机制 | `/api/environment` |
| **会话模块** | 对话管理 | 会话创建、消息存储、上下文管理 | `/api/sessions` |
| **AI模块** | 智能分析 | Prompt组装、AI调用、结果解析 | `/api/ai` |
| **文件模块** | 文件存储管理 | 上传、访问链接、删除 | `/api/files` |
| **日志模块** | 系统日志管理 | 日志查询、清理 | `/api/logs` |

### 模块依赖关系

```
用户域
├── 认证模块
│   └── 用户模块
│
├── 植物域
│   ├── 植物模块 ←── 用户模块
│   │   ├── 养护模块
│   │   ├── 诊断模块
│   │   └── 设备模块
│   │       └── 环境模块
│   │
│   └── AI域
│       ├── 会话模块 ←── 植物模块
│       │   └── AI模块 ←── 环境模块
│       └── AI模块 ←── 植物模块
│
└── 基础设施域
    ├── 文件模块（被所有模块依赖）
    └── 日志模块（被所有模块依赖）
```

## 三、分层代码架构（1分钟）

### 展示后端入口

**打开**：`backend/server/src/app.js`

**讲解词**：
> "这是后端服务的入口文件。
> 可以看到我们使用了Express框架，
> 通过中间件链处理请求。"

**关键代码片段**：

```javascript
// 中间件链
app.use(helmet());      // 安全头
app.use(cors());        // 跨域
app.use(compression()); // 压缩
app.use(express.json()); // JSON解析

// 响应格式中间件
app.use(responseMiddleware);

// 请求日志
app.use((req, res, next) => {
  logger.info(`${req.method} ${req.path}`, {
    ip: req.ip,
    body: sanitizeBody(req.body),
  });
  next();
});

// 路由注册 - 11个模块
app.use('/api/users', require('./routes/users'));
app.use('/api/plants', require('./routes/plants'));
app.use('/api/sessions', require('./routes/sessions'));
// ... 其他路由
```

### 三层架构

```
Controller层（控制器）
├── 职责：参数校验、认证检查、响应格式化
├── 示例：sessionController.js
└── 特点：薄层，不处理业务逻辑

Service层（服务层）
├── 职责：业务逻辑、事务管理、跨模块协调
├── 示例：SessionService.js
└── 特点：核心逻辑所在

Model层（模型层）
├── 职责：数据访问、关联定义
├── 示例：Session.js, Message.js
└── 特点：Sequelize ORM定义
```

### 代码示例

**Controller**（薄层）：
```javascript
// sessionController.js
async createMessage(req, res) {
  const { sessionId, content, type } = req.body;
  const result = await sessionService.createMessage(sessionId, content, type);
  res.success(result);
}
```

**Service**（核心逻辑）：
```javascript
// SessionService.js
async createMessage(sessionId, content, type) {
  // 1. 查询会话
  // 2. 组装上下文
  // 3. 调用AI
  // 4. 保存消息
  // 5. 返回结果
}
```

## 四、数据库设计（1分钟）

### 展示ER图

**打开**：`docs/02-architecture/系统架构设计.md`中的ER图

**讲解词**：
> "这是数据库的实体关系图。
> 核心实体有7个，关系清晰：
> 
> - User 拥有多个 Plant
> - Plant 绑定一个 Device
> - Plant 有多个 EnvironmentReading
> - Session 包含多个 Message
> - Message 可以产生 DiagnosisCard"

### 核心表结构

| 实体 | 表名 | 关键字段 | 关联 |
|:---|:---|:---|:---|
| User | users | user_id, wx_openid | 1:N Plant |
| Plant | plants | plant_id, user_id, current_device_id | N:1 User, 1:N Session |
| Session | sessions | session_id, user_id, plant_id | N:1 Plant, 1:N Message |
| Message | messages | message_id, session_id, role | N:1 Session |
| DiagnosisCard | diagnosis_cards | diagnosis_card_id, message_id | N:1 Message |
| Device | devices | device_id, mac_address, bound_plant_id | 1:1 Plant |
| EnvironmentReading | environment_readings | reading_id, plant_id, data_source | N:1 Plant |

### 命名规范

> "我们有一套严格的命名规范：
> - 数据库：snake_case（user_id, plant_id）
> - API：camelCase（userId, plantId）
> - 自动转换：utils/namingConverter.js"

## 五、关键技术点（1分钟）

### 1. 认证流程

**流程图**：
```
小程序 wx.login() → 获取code
    ↓
后端用code调微信接口 → 换取openid
    ↓
生成JWT Token → 返回给小程序
    ↓
后续请求携带Token → 后端验证
```

### 2. 环境数据补偿机制

**场景**：设备离线，数据缺失

**解决方案**：
```
定时任务（每2小时）
    ↓
生成ReadingTasks
    ↓
等待设备上报（5分钟容忍期）
    ↓
超时未上报 → 调用天气API补偿
    ↓
标记is_stale=true
```

### 3. AI调用流程

**诊断流程**：
```
用户上传图片
    ↓
视觉模型识别品种
    ↓
查询植物档案 + 环境数据
    ↓
组装Prompt
    ↓
语言模型诊断
    ↓
生成诊断卡片
```

**问答流程**：
```
用户提问
    ↓
查询会话历史（上下文）
    ↓
查询相关环境数据
    ↓
组装Prompt
    ↓
语言模型回答
    ↓
保存消息记录
```

## 六、技术选型说明（30秒）

### 为什么选择这些技术？

| 技术 | 选择理由 |
|:---|:---|
| **微信小程序** | 无需下载、生态丰富、适合MVP验证 |
| **Node.js** | 全栈JavaScript、开发效率高、I/O密集型场景表现好 |
| **Express** | 轻量、成熟、中间件生态丰富 |
| **Sequelize** | ORM简化数据库操作、支持迁移、关联定义清晰 |
| **MySQL** | 成熟稳定、事务支持、团队熟悉 |
| **JWT** | 无状态认证、适合分布式、小程序友好 |

### 扩展性考虑

> "当前是MVP架构，但预留了扩展性：
> - 分层设计便于微服务拆分
> - 领域划分清晰，可独立部署
> - 数据库设计支持分库分表"
