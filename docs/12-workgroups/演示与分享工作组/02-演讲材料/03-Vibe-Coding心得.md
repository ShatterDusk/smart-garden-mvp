# Vibe Coding 实战心得

> 演讲时长：3-5分钟
> 目标受众：计算机/软件专业同学
> 核心信息：AI辅助编程的效率提升、方法论、真实案例

---

## 开场（30秒）

> "大家写代码的时候，有没有遇到过这种情况：
> 一个功能明明逻辑很清楚，但就是写不出来，或者写出来一堆bug。
> 我最近尝试了一种新的开发方式——Vibe Coding，
> 让AI成为我的结对编程伙伴。这个项目从0到MVP，我只用了3周。"

---

## 一、效率提升数据（30秒）

### 时间对比估算

| 开发阶段 | 传统开发（预估） | Vibe Coding（实际） | 提升 |
|:---|:---:|:---:|:---:|
| 架构设计 | 1周 | 3天 | 60% |
| 核心功能开发 | 4周 | 1.5周 | 62% |
| 测试编写 | 2周 | 3天 | 75% |
| 文档编写 | 1周 | 2天 | 80% |
| **总计** | **8周** | **3周** | **62%** |

### 关键洞察

> "AI不会替代程序员，但会用AI的程序员会替代不会用的。"

---

## 二、AI协作模式转变（1分钟）

### 传统开发 vs Vibe Coding

| 维度 | 传统开发 | Vibe Coding |
|:---|:---|:---|
| **角色定位** | 我是实现者 | 我是架构师+Reviewer |
| **编码方式** | 手写每一行 | 写Prompt，AI生成 |
| **问题解决** | 查文档、Google | 直接问AI |
| **调试时间** | 2-4小时/bug | AI直接给解决方案 |
| **测试编写** | 痛苦、拖延 | AI生成骨架，我补充 |
| **代码质量** | 依赖个人水平 | 依赖Prompt质量 |

### 核心转变

```
写代码 → 设计代码 → 验证代码
```

> "我的主要工作从'怎么实现'变成了'要做什么、做成什么样'。
> AI负责'怎么做'，我负责'对不对'。"

---

## 三、Prompt工程实战（1.5分钟）⭐核心内容

### 有效Prompt的结构

```
背景 + 任务 + 约束 + 输出要求 + 参考示例
```

### 真实案例1：生成Service层代码

**❌ 低效Prompt**
```
帮我写一个SessionService
```

**✅ 高效Prompt**
```
请帮我实现一个SessionService，要求：

【背景】
这是一个智能园艺助手项目，使用Node.js + Express + Sequelize。

【任务】
实现会话管理功能，包括：
1. 创建会话
2. 添加消息到会话
3. 组装上下文供AI调用

【约束】
1. 使用Sequelize ORM，参考项目中已有的UserService模式
2. 错误处理使用asyncHandler包装
3. 需要包含完整的JSDoc注释
4. 方法命名使用camelCase

【输出要求】
1. 完整的Service类代码
2. 对应的单元测试（使用Jest）
3. 简要说明关键设计决策

【参考】
项目路径：backend/server/src/services/UserService.js
```

### 真实案例2：生成数据库迁移

**Prompt**
```
我需要添加一个环境数据补偿机制。

【需求】
当传感器数据缺失时，用天气API数据补偿。

【数据模型】
需要修改EnvironmentReading表，添加：
- data_source字段（sensor/weather_api/compensation）
- is_stale标记（是否为补偿数据）

【约束】
1. 已有数据需要迁移，默认data_source='sensor'
2. 使用Sequelize迁移文件格式
3. 包含回滚逻辑

请生成迁移文件和对应的Model更新。
```

### Prompt工程心得

| 技巧 | 说明 | 效果 |
|:---|:---|:---|
| **给AI角色** | "你是一位资深Node.js工程师" | 代码风格更专业 |
| **提供上下文** | 引用项目已有代码 | 保持代码一致性 |
| **明确约束** | 命名规范、错误处理方式 | 减少返工 |
| **要求解释** | "说明设计决策" | 便于Review |
| **迭代优化** | 第一轮生成→Review→补充要求 | 逐步逼近目标 |

## 四、工作流优化（1分钟）

### 对话式开发流程

```
需求理解 ──→ 写Prompt ──→ AI生成 ──→ 人工Review ──→ 测试验证
    ↑                                              │
    └────────────── 迭代优化 ←─────────────────────┘
```

### 实际工作流示例

**场景：实现设备绑定功能**

```
Round 1: 我 → AI
"帮我设计设备绑定的API接口"

Round 2: AI → 我
[生成API设计文档]

Round 3: 我 → AI
"基于这个设计，实现DeviceService，
 参考UserService的代码风格"

Round 4: AI → 我
[生成Service代码]

Round 5: 我 → AI
"补充单元测试，覆盖边界情况：
 - 设备已被绑定
 - 设备不存在
 - 植物不存在"

Round 6: AI → 我
[生成测试代码]

Round 7: 我
运行测试，发现问题，继续迭代...
```

### 关键工具

| 工具 | 用途 |
|:---|:---|
| **AI IDE（Trae/Cursor）** | 代码生成、解释、重构 |
| **Mermaid** | 架构图、流程图（让AI生成） |
| **Markdown** | 文档、需求、设计 |
| **Jest** | 测试驱动，验证AI代码 |

## 五、代码质量保障（30秒）

### AI代码 ≠ 完美代码

**常见问题**：
1. 幻觉：引用不存在的函数
2. 过时：使用废弃的API
3. 不一致：与项目风格不符
4. 边界：遗漏异常处理

### 质量保障策略

```
AI生成 → 静态检查 → 单元测试 → 集成测试 → 人工Review
```

| 检查项 | 工具 | 责任 |
|:---|:---|:---|
| 语法/风格 | ESLint + Prettier | 自动化 |
| 类型安全 | JSDoc + 手动检查 | 人+AI |
| 单元测试 | Jest | AI生成+人补充 |
| 集成测试 | Jest + 实际调用 | 人主导 |
| 架构符合度 | 人工Review | 人 |

### 数据说话

> "这个项目有640+测试用例，代码覆盖率约60%，
> 其中70%的测试是AI生成的骨架，我补充边界情况。"

## 六、关键心得总结（30秒）

### 给想尝试Vibe Coding的同学

1. **从小的、明确的任务开始**
   - 不要一上来就让AI写整个系统
   - 先尝试：写一个工具函数、一个API接口

2. **投资Prompt工程**
   - 花10分钟写一个好的Prompt，省2小时调试
   - 建立个人Prompt模板库

3. **保持批判性思维**
   - AI会自信地给出错误答案
   - 永远要验证、测试、Review

4. **文档化你的学习**
   - 记录有效的Prompt模式
   - 沉淀AI协作的最佳实践

### 一句话总结

> "Vibe Coding不是让AI代替你思考，而是让AI放大你的思考。
> 你负责'做什么'和'对不对'，AI负责'怎么做'。"
