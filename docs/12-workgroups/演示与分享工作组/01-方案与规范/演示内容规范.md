# 演示内容规范

## 演示标准结构

### 15分钟技术分享标准流程（更新版）

```
┌─────────────────────────────────────────────────────────────┐
│  0:00-1:00  开头：项目怎么来的（挑战杯流产项目）              │
│  1:00-3:00  产品演示（配合小程序界面，边看边说）              │
│  3:00-5:00  技术架构（配合Mermaid图 + 代码演示）             │
│  5:00-9:00  亮点1：AI辅助开发（重点讲）                      │
│  9:00-12:00 亮点2：GitHub二次开发（重点讲）                  │
│  12:00-14:00 亮点3：分层架构（顺带讲）                       │
│  14:00-15:00 结尾 + Q&A                                      │
└─────────────────────────────────────────────────────────────┘
```

### 核心内容（只讲3个亮点）

| 时间 | 内容 | 理由 |
|:---:|:---|:---|
| 4分钟 | AI辅助开发 | 最热点，大一最感兴趣 |
| 3分钟 | GitHub二次开发 | 实用思维，容易讲清楚 |
| 2分钟 | 分层架构 | 工程化基础，顺带提 |

**原则**：15分钟很紧，讲多不如讲精

## 演示技巧

### 开场（真实风格）

> "挑战杯定的题目，不是我的想法（笑）
> 但怎么做是我来设计的——会话怎么组织、诊断卡长什么样。
> 灵感确实是养多肉的时候来的。
> 后来挑战杯没继续参赛，但觉得投了不少设计成本，丢了可惜，就自己做下去了。"

### 讲解风格

- **像跟朋友聊天**，不是汇报工作
- **想到哪说到哪**，不用照着念
- **边看代码边说**，材料只是提示

### 过渡语（随口说）

| 场景 | 过渡语 |
|:---|:---|
| 产品→架构 | "来看看背后是怎么实现的" |
| 架构→亮点 | "聊聊几个有意思的地方" |
| 亮点1→亮点2 | "还有一个点..." |
| 亮点→结尾 | "差不多就这些，有问题随便问" |

## 风险预案

### 现场演示失败应对

| 风险 | 应对措施 |
|:---|:---|
| 小程序无法加载 | 切换到预截图 |
| 后端服务崩溃 | 展示代码+解释逻辑 |
| 数据库无数据 | 展示空状态UI |
| Mermaid图无法渲染 | 提前准备PNG备份 |

### 时间控制

- 每个环节设置时间检查点
- 超时风险时，优先保证核心内容（AI辅助+GitHub二次开发）
- 准备可删减的扩展内容
