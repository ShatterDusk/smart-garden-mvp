# 演示内容规范

## 演示标准结构

### 10-15分钟技术分享标准流程

```
┌─────────────────────────────────────────────────────────────┐
│  0:00-0:30  开场：痛点场景 + 产品演示                        │
│  0:30-2:00  产品核心功能（配合小程序界面演示）                │
│  2:00-5:00  架构讲解（配合Mermaid图 + 代码演示）             │
│  5:00-8:00  Vibe Coding实战心得（技术+方法论）               │
│  8:00-10:00 教训总结：不写文档的后果（真实案例）              │
│  10:00-12:00 软件设计心得 & AI协作方法论                      │
│  12:00-15:00 Q&A                                             │
└─────────────────────────────────────────────────────────────┘
```

## 演示技巧

### 开场钩子公式

```
痛点场景 + 具体案例 + 解决方案预告
```

示例：
> "大家有没有养过植物？我有个朋友，买了一盆多肉，两周后叶子全掉了。
> 他查百度、问朋友，得到的答案五花八门。
> 后来我做了一个小工具——拍照上传，AI告诉你'浇水过多'。
> 这就是我今天要分享的项目。"

### 代码演示原则

1. **提前定位**：不要现场找文件
2. **关键片段**：只展示核心代码，不超过20行
3. **配合讲解**：代码+口头解释，不要只念代码
4. **准备截图**：现场演示失败时的备用方案

### 过渡语模板

| 场景 | 过渡语 |
|:---|:---|
| 产品→架构 | "产品功能看完了，我们来看看背后是怎么实现的..." |
| 架构→代码 | "架构图比较抽象，我们来看具体的代码实现..." |
| 代码→心得 | "代码实现讲完了，我想分享一下用AI写代码的一些体会..." |
| 心得→教训 | "AI很强大，但也不是万能的。我踩过一些坑..." |

## 风险预案

### 现场演示失败应对

| 风险 | 应对措施 |
|:---|:---|
| 小程序无法加载 | 切换到预截图 |
| 后端服务崩溃 | 展示代码+解释逻辑 |
| 数据库无数据 | 展示空状态UI |
| Mermaid图无法渲染 | 提前准备PNG备份 |

### 时间控制

- 每个环节设置时间检查点
- 超时风险时，优先保证核心内容（架构+心得）
- 准备可删减的扩展内容
