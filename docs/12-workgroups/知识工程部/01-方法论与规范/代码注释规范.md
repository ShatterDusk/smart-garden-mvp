## 一、任务标签（Task Tags）

### 1.1 标准标签

| 标签 | 含义 | 使用场景 | 示例 |
|------|------|----------|------|
| `TODO` | 待办事项 | 功能待实现、代码待完善 | `// TODO: 添加输入参数校验` |
| `FIXME` | 需要修复 | 已知问题需要修复 | `// FIXME: 这里可能有内存泄漏` |
| `XXX` | 警告/注意 | 代码存在问题或风险 | `// XXX: 临时方案，需要优化` |
| `HACK` | 临时方案 | 不优雅的临时解决方案 | `// HACK: 为了兼容旧版本临时处理` |
| `BUG` | 已知Bug | 已确认但未修复的Bug | `// BUG: 并发情况下可能数据不一致` |
| `NOTE` | 重要说明 | 需要特别注意的代码逻辑 | `// NOTE: 这里依赖外部服务` |
| `WARNING` | 警告 | 可能导致问题的代码 | `// WARNING: 修改此配置会影响生产环境` |
| `DEPRECATED` | 已废弃 | 即将删除的旧代码 | `// DEPRECATED: 使用 newMethod() 替代` |
| `OPTIMIZE` | 待优化 | 性能或代码质量待优化 | `// OPTIMIZE: 查询效率可以优化` |
| `REFACTOR` | 待重构 | 需要重构的代码 | `// REFACTOR: 这段代码需要拆分` |

### 1.2 标签格式

```javascript
// 单行标签
// TODO: 添加用户权限校验

// 多行标签（详细说明）
/**
 * TODO: 实现缓存机制
 * - 使用 Redis 缓存热点数据
 * - 设置 5 分钟过期时间
 * - 考虑缓存穿透问题
 * @author 张三
 * @date 2026-04-11
 */

// 带优先级的标签
// TODO[HIGH]: 紧急修复支付接口超时问题
// FIXME[LOW]: 日志格式不统一
```

### 1.3 优先级标记

在标签后添加优先级，用方括号表示：

- `[CRITICAL]` - 严重，必须立即处理
- `[HIGH]` - 高优先级，尽快处理
- `[MEDIUM]` - 中等优先级，计划内处理
- `[LOW]` - 低优先级，有空时处理

## 二、JSDoc 注释规范

### 2.1 函数/方法注释

```javascript
/**
 * 函数描述
 * @param {类型} 参数名 - 参数说明
 * @param {类型} [可选参数] - 可选参数说明
 * @returns {类型} 返回值说明
 * @throws {错误类型} 错误说明
 * @example
 * // 使用示例
 * const result = functionName(param1, param2);
 */
```

### 2.2 类注释

```javascript
/**
 * 类描述
 * @class
 * @classdesc 详细描述
 * @extends 父类
 * @author 作者
 * @since 版本号
 */
class MyClass {
  /**
   * 构造函数
   * @param {Object} options - 配置选项
   */
  constructor(options) {
    // ...
  }
}
```

### 2.3 常用标签

| 标签 | 说明 | 示例 |
|------|------|------|
| `@param` | 参数说明 | `@param {string} userId - 用户ID` |
| `@returns` | 返回值 | `@returns {Promise<Object>} 用户对象` |
| `@throws` | 异常 | `@throws {Error} 当用户不存在时抛出` |
| `@deprecated` | 已废弃 | `@deprecated 使用 newMethod() 替代` |
| `@see` | 参考 | `@see https://api.docs.com` |
| `@example` | 示例 | `@example functionName('id')` |
| `@author` | 作者 | `@author 张三` |
| `@since` | 版本 | `@since 1.0.0` |
| `@ignore` | 忽略 | 不生成文档 |

## 三、代码注释规范

### 3.1 单行注释

```javascript
// 在需要解释的代码前添加注释
const result = calculate(data); // 计算结果

// 复杂的逻辑需要详细说明
// 使用二分查找提高效率，时间复杂度 O(log n)
const index = binarySearch(arr, target);
```

### 3.2 多行注释

```javascript
/*
 * 多行注释用于详细说明
 * 可以包含多行内容
 * 注意保持格式对齐
 */

/**
 * JSDoc 注释用于 API 文档
 * 会被文档生成工具解析
 */
```

### 3.3 特殊注释

```javascript
// 标记代码块开始和结束
// ==================== 初始化 ====================

// 分隔不同逻辑区域
// ----------------------------------------

// 标记待办
// TODO: 2026-04-11 添加异常处理
```

## 六、代码审查检查清单

### 6.1 注释检查

- [ ] 复杂逻辑是否有注释说明
- [ ] 公共 API 是否有 JSDoc 注释
- [ ] 临时方案是否标记 HACK/TODO
- [ ] 已知问题是否标记 FIXME/BUG
- [ ] 废弃代码是否标记 DEPRECATED

### 6.2 标签检查

- [ ] 是否有未处理的 TODO
- [ ] FIXME 是否有对应的修复计划
- [ ] XXX 是否说明了原因和解决方案
- [ ] 高优先级标签是否及时处理

## 七、最佳实践

1. **及时清理** - 完成任务后删除对应的 TODO/FIXME
2. **详细说明** - 标签后要有清晰的说明文字
3. **关联信息** - 必要时关联 Issue 或文档
4. **定期审查** - 定期扫描代码中的标签并处理
5. **团队共识** - 确保团队成员理解并遵循规范

## 八、扫描工具

### 8.1 工具位置

扫描脚本: `_dev/tools/scan-todos.js`

### 8.2 使用方法

```bash
# 扫描默认目录（backend/server/src 和 frontend）
node _dev/tools/scan-todos.js

# 扫描指定目录
node _dev/tools/scan-todos.js backend/server/src/services
```

### 8.3 扫描命令

```bash
# 扫描所有 TODO/FIXME
grep -rn "TODO\|FIXME\|XXX\|HACK" backend/server/src/
grep -rn "TODO\|FIXME\|XXX\|HACK" frontend/

# 按优先级统计
grep -rn "TODO\[CRITICAL\]\|FIXME\[CRITICAL\]" backend/server/src/
grep -rn "TODO\[HIGH\]\|FIXME\[HIGH\]" backend/server/src/
```
