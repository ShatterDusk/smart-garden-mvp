## 一、分析执行标准

### 1.1 文件扫描范围

#### 必扫文件类型
| 类型 | 扩展名 | 优先级 |
|------|--------|--------|
| JavaScript | `.js` | P0 |
| JSON配置 | `.json` | P1 |
| 数据库迁移 | `.js` (migrations/) | P1 |
| 测试文件 | `.test.js` | P2 |

#### 排除文件
```
node_modules/
dist/
build/
coverage/
*.min.js
*.bundle.js
```

### 1.2 代码行数限制
- **单次分析文件数**: ≤ 20 个
- **单次分析总行数**: ≤ 5000 行
- **复杂函数最大行数**: ≤ 100 行（超过需标记审查）

## 二、问题严重等级定义

### 🔴 高危（High）
**定义**: 可能导致安全漏洞、数据丢失或系统崩溃的问题

**示例**:
- 硬编码密钥或密码
- SQL 注入风险
- 认证绕过可能
- 未处理的 Promise 异常导致进程崩溃
- 级联删除配置错误导致数据丢失

**处理要求**: 必须立即修复，阻塞发布

### 🟠 中危（Medium）
**定义**: 影响代码质量、可维护性或存在潜在风险的问题

**示例**:
- 命名规范不一致
- 缺少错误处理
- 循环依赖
- 重复代码
- 未使用的变量/函数

**处理要求**: 在当前迭代内修复

### 🟡 低危（Low）
**定义**: 代码风格问题或轻微优化建议

**示例**:
- 注释不规范
- 变量命名不清晰
- 多余的 console.log
- 代码格式不一致

**处理要求**: 在方便时修复

## 三、分析执行流程

### 3.1 单路径分析流程

```
1. 准备阶段
   ├── 阅读该路径的「分析清单」
   ├── 确定关键文件列表
   └── 准备问题记录模板

2. 扫描阶段
   ├── 按优先级顺序读取文件
   ├── 逐条检查「分析清单」项
   └── 发现问题立即记录

3. 验证阶段
   ├── 交叉验证发现的问题
   ├── 确认问题可复现
   └── 评估影响范围

4. 报告阶段
   ├── 整理问题清单
   ├── 按严重等级排序
   └── 输出执行记录文档
```

### 3.2 问题记录规范

#### 必填字段
```yaml
问题编号: SA-{路径编号}-{序号}  # 例如: SA-4-001
严重程度: 🔴高 / 🟠中 / 🟡低
分析路径: 路径4-安全性分析
文件位置: backend/server/src/middleware/auth.js:L11
问题类型: 安全 / 架构 / 数据 / 质量
问题描述: 具体描述问题现象
风险影响: 可能造成什么后果
修复建议: 具体的修复方案
状态: 待修复 / 修复中 / 已修复
```

#### 可选字段
```yaml
复现步骤: 如何复现该问题
参考文档: 相关链接或文档
修复人: 负责修复的人员
修复日期: YYYY-MM-DD
验证人: 验证修复的人员
```

## 四、命名规范

### 4.1 文件命名
```
# 执行记录
路径{编号}-{简短描述}.md
例如: 路径4-安全性分析.md

# 报告文件
{日期}-{类型}报告.md
例如: 20260408-问题汇总报告.md
```

### 4.2 问题编号规则
```
SA-{路径编号}-{三位序号}

示例:
SA-4-001   # 路径4的第1个问题
SA-2-015   # 路径2的第15个问题
```

## 六、工具使用规范

### 6.1 推荐工具

| 工具 | 用途 | 使用场景 |
|------|------|---------|
| ESLint | 代码规范检查 | 全项目扫描 |
| grep/ripgrep | 文本搜索 | 查找特定模式 |
| VS Code 搜索 | 跨文件查找 | 查找引用关系 |
| Git Blame | 追溯代码历史 | 问题归因 |

### 6.2 常用搜索模式

```bash
# 查找硬编码密钥
grep -r "password\|secret\|key" --include="*.js" .

# 查找 console.log
grep -r "console\.log\|console\.error" --include="*.js" .

# 查找 TODO/FIXME
grep -r "TODO\|FIXME\|XXX" --include="*.js" .

# 查找未使用的导出（需配合工具）
```

## 七、协作规范

### 7.1 每日同步
- 更新问题清单状态
- 标记已修复的问题
- 记录阻塞问题

### 7.2 问题升级
当发现以下情况时，立即升级：
- 发现可能导致数据丢失的 bug
- 发现安全漏洞
- 发现架构设计缺陷
- 发现大量同类问题（>10个）

### 7.3 文档更新
- 发现问题后立即更新执行记录
- 修复问题后更新问题清单状态
- 每天结束时更新跟踪报告
