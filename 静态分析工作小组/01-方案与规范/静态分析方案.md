## 一、分析目标

对 PlantGPT 项目进行系统性静态代码分析，识别以下类别的问题：

1. **安全问题**: 硬编码密钥、认证绕过、SQL注入风险
2. **架构问题**: 层间调用不规范、循环依赖、职责混乱
3. **数据问题**: 模型关联错误、命名转换遗漏、字段映射不一致
4. **质量问题**: 错误处理缺失、日志不规范、硬编码值

## 二、6条分析路径详解

### 🔵 路径1: 前端数据流与API调用链分析

**目标**: 追踪从前端页面 → API 调用 → 数据展示的完整数据流

#### 关键入口文件
| 文件 | 作用 | 行数 |
|------|------|------|
| `frontend/app.js` | 应用入口，globalData 初始化 | 55 |
| `frontend/utils/api.js` | 统一请求封装 | 605 |
| `frontend/utils/config.js` | 环境配置 | - |

#### 页面级分析节点（14个页面）
```
pages/
├── index/          ← 首页（植物列表入口）
├── login/          ← 登录页（认证起点）
├── plants/         ← 植物列表
├── plant-detail/   ← 植物详情（聚合查询重灾区）
├── add-plant/      ← 添加植物
├── sessions/       ← 会话列表
├── plant-sessions/ ← 植物会话
├── qna/            ← AI问答
├── quick-analyze/  ← 快速诊断
├── device-manage/  ← 设备管理
├── device-detail/  ← 设备详情
├── metric-detail/  ← 指标详情
├── weather/        ← 天气页
└── care-records/   ← 养护记录（隐含在plant-detail中）
```

#### 分析清单
- [ ] **API 调用覆盖率**：每个页面调用了哪些 api.js 方法？是否有未使用的导出函数？
- [ ] **Promise 链深度**：检查回调地狱，特别是 `uploadImage()` 的嵌套 Promise
- [ ] **Token 生命周期**：`getToken()` → `setToken()` → `clearToken()` 的调用点是否完整？
- [ ] **错误处理一致性**：所有 `.catch()` 是否都有用户提示？
- [ ] **硬编码检测**：URL路径、状态码等是否直接硬编码而非引用常量？

### 🟢 路径2: 后端请求处理全链路分析

**目标**: 验证 Route → Controller → Service → Model 的完整调用链

#### 架构层次
```
app.js (路由注册)
  ↓
routes/*.js (12个路由文件)
  ↓
controllers/*.js (11个控制器)
  ↓
services/*.js (9个服务 + BaseService)
  ↓
models/*.js (13个模型)
```

#### 关键链路示例（以植物模块为例）
```
Client → GET /api/plants
    → routes/plants.js
    → plantController.getPlants()
    → PlantService.getPlantList()
    → Plant.findAndCountAll()
    → PlantService.getPlantDevices() + getLatestDiagnoses()
    → Controller 格式化响应
    → Client
```

#### 分析清单
- [ ] **路由-控制器映射**：`app.js` 注册的12个路由是否都有对应控制器？
- [ ] **控制器-服务调用**：每个 controller 方法是否都通过 service 层访问数据库？
- [ ] **Service 层依赖**：`services/index.js` 导出的9个 service 是否都被使用？
- [ ] **BaseService 继承**：各 Service 是否正确继承 BaseService？
- [ ] **循环依赖检测**：Service 之间是否存在循环引用？

### 🟠 路径3: 数据模型与关联完整性分析

**目标**: 确保 Model 定义、外键约束、关联配置三者一致

#### 模型清单（13个）
| 模型文件 | 主表 | 关联表 | 优先级 |
|---------|------|--------|--------|
| User.js | users | plants, sessions, devices | ⭐⭐⭐ |
| Plant.js | plants | diagnosis_cards, care_records, sessions | ⭐⭐⭐ |
| Device.js | devices | plants, environment_readings | ⭐⭐ |
| Session.js | sessions | messages, diagnosis_cards, plants | ⭐⭐⭐ |
| Message.js | messages | sessions, diagnosis_cards | ⭐⭐ |
| DiagnosisCard.js | diagnosis_cards | sessions, messages, plants | ⭐⭐ |
| CareRecord.js | care_records | plants | ⭐ |
| EnvironmentReading.js | environment_readings | environment_reading_values, devices | ⭐⭐ |
| EnvironmentReadingValue.js | environment_reading_values | environment_readings, metrics | ⭐ |
| EnvironmentMetric.js | environment_metrics | - | ⭐ |
| ReadingTask.js | reading_tasks | - | ⭐ |
| UserConfig.js | user_configs | users | ⭐ |

#### 关键配置文件
- `models/associations.config.js` - 级联策略定义
- `models/index.js` - 模型导入与关联注册

#### 分析清单
- [ ] **外键一致性**：Model 中定义的 `foreignKey` 是否与数据库实际外键一致？
- [ ] **级联删除风险**：`CASCADE` 配置的关联是否会导致意外批量删除？
- [ ] **关联定义覆盖**：`belongsTo` / `hasMany` / `belongsToMany` 是否完整注册？
- [ ] **字段命名**：数据库 snake_case 与 Sequelize camelCase 映射是否正确？
- [ ] **迁移文件同步**：迁移文件是否与 Model 定义一致？

### 🔴 路径4: 安全性与敏感信息分析

**目标**: 发现安全漏洞、硬编码密钥、权限绕过等问题

#### 关键安全节点
| 文件 | 安全关注点 | 风险等级 |
|------|-----------|---------|
| `middleware/auth.js` | JWT_SECRET 默认值 | 🔴 高 |
| `middleware/deviceAuth.js` | 设备认证逻辑 | 🔴 高 |
| `controllers/cosController.js` | COS 签名生成 | 🟠 中 |
| `config/ai.js` | AI API Key | 🔴 高 |
| `config/environment.js` | 第三方 API Key | 🔴 高 |
| `app.js` | 敏感字段过滤 | 🟡 低 |

#### 分析清单
- [ ] **硬编码密钥检测**：`JWT_SECRET` 是否存在默认值 `'your-secret-key'`？
- [ ] **Token 传输安全**：前端是否始终使用 HTTPS？
- [ ] **认证中间件覆盖**：哪些路由缺少 `authMiddleware` 保护？
- [ ] **用户权限隔离**：`req.user.userId` 是否在所有数据查询中作为过滤条件？
- [ ] **SQL 注入风险**：Sequelize 查询是否使用参数化查询？
- [ ] **COS 签名安全**：上传签名是否有时效限制？是否校验文件类型？

### 🟣 路径5: 命名规范与转换一致性分析

**目标**: 确保 snake_case ↔ camelCase 转换无遗漏

#### 命名转换架构
```
数据库 (snake_case)
    ↓ namingConverter.js
后端 API (camelCase)
    ↓ api.js 响应
前端变量 (camelCase)
```

#### 关键转换点
| 位置 | 转换方式 | 示例 |
|------|---------|------|
| `utils/namingConverter.js` | 工具函数 | `toCamelCase()` / `toSnakeCase()` |
| `PlantService.js` | 手动映射 | `fieldMapping` 对象 |
| `utils/formatters.js` | 格式化工具 | `formatDiagnosisCards()` |
| Controller 响应 | 手动构造 | 如 `plantController.js` |

#### 分析清单
- [ ] **自动 vs 手动转换混用**：是否同时存在自动转换和手动 fieldMapping？
- [ ] **Controller 层字段名**：响应中的字段是否全部为 camelCase？
- [ ] **前端接收字段**：页面 JS 中使用的字段名是否与后端响应一致？
- [ ] **枚举值一致性**：`plantCategory`、`actionType`、`data_source` 等枚举前后端是否统一？

### ⚫ 路径6: 错误处理与日志规范分析

**目标**: 统一错误处理模式，避免信息泄露

#### 错误处理架构
```
请求异常
    ↓
Service 层 throw err
    ↓
Controller 层 catch → error(res, message, statusCode)
    ↓
errorHandler.js 统一处理未捕获异常
    ↓
响应 { code, message, data }
```

#### 关键文件
| 文件 | 作用 | 分析重点 |
|------|------|---------|
| `middleware/errorHandler.js` | 全局错误中间件 | 是否泄露堆栈信息？ |
| `middleware/response.js` | 响应格式化 | success/error 格式是否统一？ |
| `utils/logger.js` | 日志工具 | 日志级别是否合理？ |
| `middleware/validator.js` | 请求校验 | 校验失败响应格式 |

#### 分析清单
- [ ] **错误码规范**：是否使用统一的错误码体系？
- [ ] **生产环境堆栈**：`errorHandler.js` 在生产环境是否隐藏 `err.stack`？
- [ ] **日志注入**：日志中记录的 `req.body` 是否经过脱敏？
- [ ] **console.log 残留**：代码中是否还有 `console.log` / `console.error` 未替换为 `logger`？
- [ ] **Promise 异常捕获**：所有 async 函数是否都有 try-catch 或 `.catch()`？
